# ECC 부분 설치 상세 가이드 (2026-03-06)

## 문서 목적
- `everything-claude-code`를 interStella에 그대로 설치하지 않고, 프로젝트에 유효한 부분만 가져온 이유와 운영법을 정리한다.
- 이 문서는 실무 적용용이며, "왜 이걸 깔았는지"와 "어떻게 쓰는지"를 바로 확인할 수 있도록 작성한다.

## 배경: 왜 전체 설치가 아니라 부분 설치인가
- interStella의 핵심은 `Unity 2022 + C# + FishNet + Steam` 검증 루프다.
- 원본 레포는 범용 생산성 규칙이 강점이지만, 언어/플러그인 중심 구성이 Unity 네트워크 검증 요구와 1:1로 맞지 않는다.
- 그래서 "생산성 템플릿"은 취하고, "프로젝트와 충돌할 수 있는 광범위 규칙"은 제외했다.

핵심 판단:
- 채택: 작업 루프 표준화, 완료 정의(DoD), 기록 포맷, 실행 스니펫
- 제외: interStella 맥락과 직접 관련 없는 언어/도구 일반 규칙

## 이번에 실제로 설치한 항목
1. `.codex/config.toml`
- 목적: Codex 세션의 기본 행동을 interStella 프로젝트 방식으로 고정
- 포함된 핵심 설정:
  - 작업 루프: `탐색 -> 최소 변경 구현 -> 검증 -> 문서 누적 -> 보고`
  - 네트워크 DoD: `Owner 입력/Remote 관찰/재접속 슬롯 복구/durable-transient 경계`
  - 문서 규칙: `기획서/TASK/Build Guide` Append-Only 누적

2. `Docs/Codex Workflow Pack.md`
- 목적: 팀이 바로 재사용할 수 있는 운영 가이드 문서화
- 포함된 핵심 내용:
  - 부분 설치 원칙
  - 네트워크 완료 기준
  - 2프로세스 Host/Client 실행 힌트
  - 로그 확인 패턴(`Assigned`, `Released`)

3. 운영 이력 누적
- `Docs/기획서.md`, `Docs/TASK.md`, `Docs/Build Guide.md`에 2026-03-06 11:42(KST) 이력 추가
- 이유: 무엇을 왜 적용했는지 추적 가능하게 유지

## 왜 설치했는가: 문제와 해결
문제 1. 세션마다 작업 품질이 흔들림
- 해결: `.codex/config.toml`로 루프/완료 기준을 기본값으로 고정

문제 2. 네트워크 작업이 로컬 스모크에서 끝나 검증 누락 발생
- 해결: DoD에 `Owner/Remote/Reconnect`를 필수 조건으로 고정

문제 3. 문서 기록이 사람 습관에 의존
- 해결: Append-Only 기록 대상을 명시하고 같은 포맷으로 누적

문제 4. 다음 작업자에게 맥락 전달 비용이 큼
- 해결: `Codex Workflow Pack`에 운영 규칙과 실행 스니펫을 집약

## 활용법 (실전)
### 1) 작업 시작 전
- `Docs/Codex Workflow Pack.md`의 DoD를 먼저 확인한다.
- 변경이 네트워크 영향인지 판단한다.

### 2) 작업 중
- 반드시 작은 단위로 변경하고, 각 단위마다 검증 로그를 확보한다.
- 네트워크 영향 변경이면 아래 3가지를 확인한다:
  - Owner만 입력되는가
  - Remote는 관찰만 하는가
  - 연결 해제/재접속 시 슬롯이 회수/재할당되는가

### 3) 작업 완료 직전
- durable state와 transient event 경계를 체크한다.
- 상태 튐/중복 이벤트/누락 이벤트가 없는지 확인한다.

### 4) 기록
- `기획서/TASK/Build Guide`에 같은 타임스탬프로 Append한다.
- 권장 기록 3줄:
  - 변경 이유
  - 검증 결과
  - 다음 리스크/다음 작업

## 범위와 한계
- 이 부분 설치는 "작업 운영 품질"을 올리는 장치다.
- 게임 기능을 직접 추가하거나 자동으로 버그를 고치지는 않는다.
- 따라서 실제 가치가 나오려면, 위 체크리스트를 매 작업에 실제로 적용해야 한다.
